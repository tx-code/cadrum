# Boolean 演算は単体×単体のみ公開する方針

## 実測 (OCCT 8.0.0, cadrum HEAD)

### 1. union: グループ内自己交差は融合されない

`test_union_cylinders` の出力:

```
union([A], [B])         solid count: 1
union([A, B], [C, D])   solid count: 4
union([ABCD], [ABCD])   solid count: 4
```

A,B,C,D は r=1.1 で互いに距離 √2〜2.0 なので全て交差している。それでも 4 個に分かれて返る。args グループ内 (A-B) と tools グループ内 (C-D) の交差は融合対象外で、args↔tools 間の交差のみ処理されるため。

公式ドキュメント (Boolean Operations User Guide):

> SOLIDs inside the COMPOUND must not contact (intersect or touch) each other.

OCCT 開発者 Mikhail Sazonov の発言 (Forum: Merge body with Fuse function):

> General fuse algorithm just splits the shells of the arguments at intersections and gathers all in one compound, which is not a manifold solid.

8.0.0 でもこの仕様は変わっていない (Boolean 周りの変更は collection 型の重複コンテナ除去フィルタなど、本件と無関係)。

### 2. subtract: tools 内自己交差で体積が破綻

`test_subtract_sphere_with_multiple_holes` (球 r=5 から X/Y/Z 軸方向の r=1 円柱穴を抜く):

| 方式 | count | volume |
|---|---|---|
| `[sphere].subtract(&[hole_x, hole_y, hole_z])` | 1 | **430.30** |
| 逐次 (X→Y→Z) | 1 | **441.61** |

期待値の算出:
- V(sphere) = 4/3 π·5³ ≈ 523.598
- V(cyl ∩ sphere) ≈ 31.10 (1本)
- V(bicylinder Steinmetz inside sphere) ≈ 5.33
- V(tricylinder Steinmetz inside sphere) ≈ 4.69
- V(∪3本 inside sphere) by 包除原理 = 3·31.10 − 3·5.33 + 4.69 ≈ 81.93
- 期待: 523.60 − 81.93 = **441.67** ← 逐次と一致

multi-tools が引いた量 = 523.60 − 430.30 = 93.30 = **3·V(cyl∩sphere) = Σ V(tool_i)**。
つまり **重なり領域 (bicyl/tricyl) を 2 重・3 重カウントしている**。count=1 で「正常な solid」に見えるが体積は物理的に意味を持たない。

### 3. intersect: multi-tools は集合演算にならない

`test_intersect_sphere_with_multiple_cylinders` (球 r=5 と r=0.8 円柱 3本):

| 方式 | count | volume |
|---|---|---|
| `[sphere].intersect(&[cyl_x, cyl_y, cyl_z])` | **3** | 59.93 |
| 各 `sphere.intersect(&cyl_i)` 単独 (3個合算) | 3 | 19.98 × 3 = 59.93 |
| 逐次 `sphere ∩ cyl_x ∩ cyl_y ∩ cyl_z` | 1 | 2.40 |

ユーザーが期待しがちな:
- (a) `sphere ∩ (cyl_x ∪ cyl_y ∪ cyl_z)` (ウニ胴体形状、連結 1 個)
- (b) `sphere ∩ cyl_x ∩ cyl_y ∩ cyl_z` (中心の小領域、1 個)

の **どちらでもない**。multi-tools intersect は **各 tool ごとの sphere∩tool_i を別 Solid として返す**だけで、集合演算としての intersect の意味論を持たない。

---

## なぜ OCCT がこういう設計なのか

General BOP (`BOPAlgo_Builder`) の本質:

> The General Fuse operation does not perform any other modification than splitting the input shapes based on their intersection information, and the result is a compound containing all split parts of the arguments.

つまり「複数入力を相互交差で分割するだけ」のアルゴリズムで、`Fuse`/`Cut`/`Common` は分割後のピースをどう選別するかの違いに過ぎない。**入力ピースの数 ≒ 出力 Solid の数** という性質はこの設計に由来する。集合論的な op(∪args, ∪tools) を期待する API ではない。

`SimplifyResult()` / `ShapeUpgrade_UnifySameDomain` は結果のフェース統合のみで、Solid 数を減らさない。`BOPAlgo_CellsBuilder::RemoveInternalBoundaries()` はマテリアル指定が必要でユースケースが限定的。**直接的に「集合論的 union/cut/common」を提供する OCCT API は存在しない**。

---

## 他の OCCT ラッパーはどうしているか

仮説: 同じ OCCT を使う CadQuery (Python) は回避策を持っているのではないか?

→ **持っていない**。`cadquery/occ_impl/shapes.py` の実装は cadrum と完全に同じ呼び方をしている:

```python
def _bool_op(self, args, tools, op, parallel=True):
    arg = TopTools_ListOfShape()
    for obj in args:
        arg.Append(obj.wrapped)
    tool = TopTools_ListOfShape()
    for obj in tools:
        tool.Append(obj.wrapped)
    op.SetArguments(arg)
    op.SetTools(tool)
    op.SetRunParallel(parallel)
    op.Build()
    return Shape.cast(op.Shape())

def cut(self, *toCut, tol=None):
    return self._bool_op((self,), toCut, BRepAlgoAPI_Cut())

def fuse(self, *toFuse, glue=False, tol=None):
    return self._bool_op((self,), toFuse, BRepAlgoAPI_Fuse())

def intersect(self, *toIntersect, tol=None):
    return self._bool_op((self,), toIntersect, BRepAlgoAPI_Common())
```

確認できた事実:

- **シグネチャは単一×複数**: `args=(self,)` 固定、`tools=*toFuse` 可変長。tools 同士の交差は OCCT に丸投げ。
- **pairwise fold もしない、UnifySameDomain で後処理もしない**。`clean=True` オプションは `ShapeUpgrade_UnifySameDomain` を呼ぶが、これは同一サーフェス上のフェース統合のみで体積の多重カウントは直さない。
- `Workplane.combine()` も最終的に `s.fuse(*items)` で一括 fuse するため同じ問題が出る。
- **したがって CadQuery で `sphere.cut(hole_x, hole_y, hole_z)` を tools 自己交差ありで呼ぶと cadrum で観測した 430.30 と同じ間違った体積が出る**。

含意:

- 「定評ある CadQuery がこの形で N 年動いている」= 「tools 同士の交差を踏まないユースケースが実用上多い」もしくは「ユーザーが気付かず体積誤差を許容している」。前者の前提に依存する API は cadrum が継承する価値が薄い。
- cadrum が単体×単体に絞れば、CadQuery には無い「OCCT の罠を型レベルで封じる」ラッパーになる。
