# OCCT wasm32ビルド方針

## 目標

```
make -C wasm-experiment/ generate
```

が通ること。これは以下を意味する:

1. chijin が `wasm32-unknown-unknown` ターゲットでライブラリとしてコンパイルできる
2. アプリケーション側が `chijin::Solid`, `chijin::Boolean`, `chijin::Shape` 等を普通に使える
3. wasm-bindgen でJS関数としてエクスポートできる
4. wasm-pack が .wasm + JS glue + .d.ts を生成できる

## 意義

chijin自体をwasm化するのではなく、**chijinを使ったアプリケーションが
`wasm32-unknown-unknown` でビルドできる**ことがゴール。

wasm-bindgenを付けるのはアプリケーション側の責任であり、
chijinはただのライブラリとしてwasmターゲットでリンクが通ればよい。

### wasm-experiment の構成

```
wasm-experiment/
  Cargo.toml        # chijin = { path = ".." }, wasm-bindgen = "*"
  makefile           # generate: wasm-pack build
  src/
    lib.rs           # #[wasm_bindgen] pub fn chijin_svg() -> String
    chijin.rs        # examples/chijin.rs のコピー (chijin() -> Solid)
```

アプリ側 (`src/lib.rs`):
```rust
use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn chijin_svg() -> String {
    chijin::chijin().to_svg(DVec3::new(1.0, 1.0, 1.0), 0.5).unwrap()
}
```

JS側:
```js
import init, { chijin_svg } from './pkg/wasm_experiment.js';
await init();
document.getElementById('svg').innerHTML = chijin_svg();
```

この例では `Solid::cylinder`, `Boolean::union`, `Boolean::subtract`,
`Shape::to_svg` が全て wasm 上で動作する必要がある。

## 現状

- `src/lib.rs` — cfgゲートなし。全モジュールが常にコンパイル対象。
- `build/main.rs` — wasm32分岐あり（libcxx::build 後に todo!）。bundled/prebuiltは動作済み。
- `build/libcxx.rs` — libcxxダウンロード・ビルドのスケルトン実装済み。

wasm32ターゲットでビルドすると、libcxxビルド後にtodo!で停止する。
libcxxビルドにはC標準ライブラリ（libc）のヘッダが必要であり、
wasm32-unknown-unknownにはlibcが存在しないことが判明。

## なぜ可能か

chijinが使うOCCTの機能は本質的にメモリ演算で閉じている:

| 機能 | OS依存 | 備考 |
|------|--------|------|
| Shape構築 (Box, Cylinder等) | なし | 純粋なメモリ演算 |
| Boolean演算 (Fuse, Cut, Common) | なし | 純粋なメモリ演算 |
| Meshing | なし | 純粋なメモリ演算 |
| STEP/BREP I/O | なし | RustReader/RustWriter経由で既にファイル不要 |
| SVG/HLR投影 | なし | 純粋なメモリ演算 |

OS依存はTKernelのOSD層に集中しており、chijinの機能パスからは到達しない。

## C/C++標準ライブラリ戦略

### 問題

`wasm32-unknown-unknown` にはC標準ライブラリ (libc) もC++標準ライブラリ (libc++) も
存在しない。OCCTとcxxクレートのコンパイルには両方が必要。

libcxx を単体でビルドしようとすると、libcxx 自身が libc のヘッダ群
(`math.h`, `stdlib.h`, `string.h`, `time.h`, `wchar.h`, `stdio.h`, ...)
を前提としており、これらを全て自作スタブで用意するのは終わりのない作業になる。

### 解決策: wasi-libc (top-half) + libcxx

[WebAssembly/wasi-libc](https://github.com/WebAssembly/wasi-libc) は内部が2層構造:

```
wasi-libc/
  libc-top-half/     ← musl ベース。純粋な演算で閉じる関数群
    musl/              math (sin, cos, ...), string (memcpy, strlen, ...),
                       stdlib (malloc/free via dlmalloc), etc.
  libc-bottom-half/  ← WASI syscall ラッパー。OS依存
    cloudlibc/src/     clock_gettime, getrusage, fd_read, fd_write, etc.
    sources/           WASI API 呼び出しの実装
```

**top-half だけを `wasm32-unknown-unknown` 向けにビルド**すれば、
libc の完全なヘッダ群と `.a` が得られる:

- `sin()`, `cos()`, `sqrt()` 等 → musl の純粋な数学実装（正確）
- `malloc()`, `free()` → dlmalloc（動的メモリ確保が動く）
- `memcpy()`, `strlen()`, `strcmp()` 等 → musl の文字列操作
- `snprintf()` 等 → musl のフォーマット関数

bottom-half が提供するはずのシンボル (`clock_gettime`, `open`, `read` 等) は
リンク時に空スタブで埋める。chijin の機能パスからは到達しないため実害なし。

この libc の上に libcxx をビルドすることで、libcxx が要求する C ヘッダが
全て wasi-libc の top-half から供給され、自作スタブは一切不要になる。

### ビルドの流れ

```
1. wasi-libc top-half をビルド
   → libc ヘッダ群 + libc.a (math, string, malloc, ...)

2. libcxx をビルド (top-half の libc ヘッダを参照)
   → C++標準ヘッダ群 + libc++.a

3. OCCT をビルド (libc ヘッダ + libcxx ヘッダを参照)
   → OCCT 静的ライブラリ群

4. cxx_build (libc ヘッダ + libcxx ヘッダを参照)
   → Rust-C++ ブリッジ
```

### スタック全体像

```
[JS] wasm-bindgen
  ↓
[Rust] アプリケーション (wasm-experiment)
  ↓ use chijin::*
[Rust] chijin (ライブラリ)
  ↓ cxx FFI
[C++] OCCT (静的ライブラリ、OS依存部スタブ化)
  ↓
[C++] libc++ (LLVM libcxx、OS機能無効)
  ↓
[C]   wasi-libc top-half (musl math + dlmalloc + string)
  ↓   bottom-half シンボルは空スタブ
wasm linear memory
```

## 5つの作業

### 作業1: build.rs の分割 ✅

`build.rs` を `build/` ディレクトリに分割済み。

```
build/
  main.rs        # エントリポイント、ターゲット分岐
  occt.rs        # OCCTダウンロード・CMakeビルド・パッチ
  libcxx.rs      # libcxxダウンロード・ビルド
  link.rs        # リンク設定・cxx_build
  stub.rs        # stub_all_top_level_bodies 等
```

`cargo test` 通過確認済み。

### 作業2: wasi-libc (top-half) のwasm向けビルド

wasi-libc リポジトリをダウンロードし、top-half のみを
`wasm32-unknown-unknown` 向けにビルドする。

ビルド成果物:
- `sysroot/include/` — C標準ヘッダ群 (`math.h`, `string.h`, `stdlib.h`, ...)
- `sysroot/lib/libc.a` — C標準ライブラリ本体 (musl math + dlmalloc)

bottom-half が提供するシンボルのうち、リンク時に未解決になるものは
空スタブ (`.c` ファイル) を用意してコンパイル・リンクする。

`build/wasi_libc.rs` に実装する。

### 作業3: libc++ (libcxx) のwasm向けビルド

作業2の wasi-libc ヘッダを `CMAKE_SYSROOT` で供給し、
libcxx を `wasm32-unknown-unknown` 向けにクロスコンパイルする。

必要なダウンロード (LLVM 18.1.8):
- `libcxx-18.1.8.src.tar.xz` (~4MB) — libcxx ソース
- `cmake-18.1.8.src.tar.xz` (~9KB) — LLVM cmake モジュール
- `runtimes-18.1.8.src.tar.xz` (~7KB) — runtimes cmake モジュール

3つの cmake モジュールを1ディレクトリに統合して `CMAKE_MODULE_PATH` で渡す。

```cmake
cmake -G "MinGW Makefiles" libcxx-18.1.8.src \
  -DCMAKE_SYSTEM_NAME=Generic \
  -DCMAKE_SYSROOT=<wasi-libc-sysroot> \
  -DCMAKE_C_COMPILER=clang \
  -DCMAKE_CXX_COMPILER=clang++ \
  -DCMAKE_C_COMPILER_TARGET=wasm32-unknown-unknown \
  -DCMAKE_CXX_COMPILER_TARGET=wasm32-unknown-unknown \
  -DCMAKE_AR=llvm-ar \
  -DCMAKE_RANLIB=llvm-ranlib \
  -DCMAKE_TRY_COMPILE_TARGET_TYPE=STATIC_LIBRARY \
  -DCMAKE_CXX_FLAGS="-fno-exceptions -fno-rtti" \
  -DCMAKE_MODULE_PATH=<all-cmake-modules> \
  -DLIBCXX_ENABLE_SHARED=OFF \
  -DLIBCXX_ENABLE_STATIC=ON \
  -DLIBCXX_ENABLE_FILESYSTEM=OFF \
  -DLIBCXX_ENABLE_THREADS=OFF \
  -DLIBCXX_ENABLE_MONOTONIC_CLOCK=OFF \
  -DLIBCXX_ENABLE_RANDOM_DEVICE=OFF \
  -DLIBCXX_ENABLE_LOCALIZATION=OFF \
  -DLIBCXX_ENABLE_EXCEPTIONS=OFF \
  -DLIBCXX_CXX_ABI=none \
  -DLIBCXX_INCLUDE_BENCHMARKS=OFF
```

無効化の理由:
- `FILESYSTEM=OFF` — wasmにファイルシステムなし
- `THREADS=OFF` — wasmはシングルスレッド
- `MONOTONIC_CLOCK=OFF` — `clock_gettime` なし
- `RANDOM_DEVICE=OFF` — `/dev/urandom` なし
- `LOCALIZATION=OFF` — ロケール不要、POSIX依存排除
- `EXCEPTIONS=OFF` — wasm例外は未成熟
- `CXX_ABI=none` — libcxxabiを使わない
- `INCLUDE_BENCHMARKS=OFF` — google-benchmark は不要

libcxxabi は不要。`__cxa_*` シンボルは以下で対処:
- `__cxa_throw` 等 → `-fno-exceptions` で発生しない
- `typeinfo` → `-fno-rtti` で発生しない
- `__cxa_guard_acquire/release` → シングルスレッドなので単純スタブ
- `__cxa_atexit` → スタブ
- `__cxa_pure_virtual` → `abort` で実装

ビルド成果物:
- `include/c++/v1/` — C++標準ヘッダ群（OCCTは `-I` 指定、cxx_buildは `.include()` 指定）
- `lib/libc++.a` — C++標準ライブラリ本体（静的リンク）

`build/libcxx.rs` に実装する（既存スケルトンを wasi-libc 連携に更新）。

### 作業4: OCCTソースのスタブ化

build.rsの既存 `patch_occt_sources` + `stub_all_top_level_bodies` を拡張。
OS依存ファイルのメソッドボディを空化する。

スタブ対象 (TKernel OSD層):

| ファイル | 依存 | 方針 |
|---------|------|------|
| `OSD_File.cxx` | open/read/write/close | 空化 |
| `OSD_Path.cxx` | ファイルパス操作 | 空化 |
| `OSD_Signal.cxx` | signal, MessageBoxA | 空化 |
| `OSD_SharedLibrary.cxx` | dlopen/LoadLibrary | 空化 |
| `OSD_Thread.cxx` | pthread_create | 空化 |
| `OSD_Environment.cxx` | getenv/setenv | 空化 |
| `OSD_Directory.cxx` | mkdir/opendir | 空化 |
| `OSD_FileSystem.cxx` | fopen/fclose | 空化 |
| `OSD_Parallel.cxx` | TBB/OpenMP | 空化 |
| `OSD_MemInfo.cxx` | /proc/meminfo, sysinfo | 空化 |
| `Standard_ErrorHandler.cxx` | setjmp/longjmp | 空化 |
| `FSD_File.cxx` | ファイルI/O | 空化 |
| `Plugin_MapOfFunctions.cxx` | 動的ロード | 空化 |

スタブ不要 (メモリ演算のみ):
- `Standard_MMgrOpt.cxx` — malloc/freeラッパー → wasi-libc の dlmalloc で動作
- `NCollection_*` — コンテナ・アルゴリズム
- `gp_*`, `TopLoc_*`, `TopoDS_*` — 幾何データ構造

手順:
1. `cargo build --target wasm32-unknown-unknown` でリンクエラー収集
2. 未解決シンボルからソースファイル特定
3. `patch_occt_sources` に追加して空化
4. リンクエラーが消えるまで繰り返し

### 作業5: build.rs のwasm向けCMake設定

`build/occt.rs` に `build_wasm` 関数を追加し、
OCCTをwasm向けにCMakeビルドする。作業2〜4をここで統合する。

wasm時のCMake設定差分:
```cmake
-DCMAKE_SYSTEM_NAME=Generic
-DCMAKE_SYSROOT=<wasi-libc-sysroot>
-DCMAKE_C_COMPILER=clang
-DCMAKE_CXX_COMPILER=clang++
-DCMAKE_C_COMPILER_TARGET=wasm32-unknown-unknown
-DCMAKE_CXX_COMPILER_TARGET=wasm32-unknown-unknown
-DCMAKE_CXX_FLAGS="-fno-exceptions -fno-rtti -I<libcxx>/include"
```

## リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| wasi-libc top-half のみのビルドが困難 | ビルドエラー | CMakeLists.txt を読んで対象ファイル特定 |
| bottom-half スタブの不足 | リンクエラー | エラー駆動で追加 |
| OCCTのスタブ化漏れ | リンクエラー | エラー駆動で追加 |
| OCCTが内部でファイルパスを仮定 | 実行時エラー | stream I/O経路のみ使用 |
| wasmバイナリサイズ | 数MB超 | wasm-opt + LTO + 未使用コード除去 |
| STEP readerのスキーマファイル参照 | 読み込み失敗 | STEPCAFControl_Readerの内部パスを調査 |

## 成功基準

```
make -C wasm-experiment/ generate
```

が通り、生成された .wasm をブラウザで実行して
chijin (太鼓) のSVGが描画されること。
