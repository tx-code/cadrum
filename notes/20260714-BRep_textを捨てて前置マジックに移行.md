# BRep text を捨てて、カラートレイラーを前置マジックに移行

`notes/20260406-BRep末尾カラートレイラー方式.md` の「棚上げ事項: ストリーム位置によるトレイラー回避」を回収した。棚上げの理由そのものが text だったので、text を捨てて解決した。

## 形式

```
旧: [ペイロード][N x entry][u32 N][magic "CDCL"]   ← 末尾4バイトを検索
新: [ペイロード][magic "CDCL"][u32 N][N x entry]   ← ペイロード終端で決め打ち
```

## なぜ text だと前置マジックにできなかったか

前置マジックには「BRep のパースがどこで終わったか」が要る。実測:

- **binary**: `BinTools::Read` はペイロードをちょうど使い切る(パース可能な最小 prefix = ペイロード長)。末尾にゴミが付いていても読める。`istringstream` の `tellg()` で終端が取れる。
- **text**: `BRepTools::Read` は `operator>>` ベースで、ペイロードが空白で終わる(末尾 `"*\n\n+1 0 "`)。最後の空白を消費したか確定せず、終端が ±1 バイト曖昧。さらに truncate すると **アクセス違反でプロセスごと落ちる**(`Standard_Failure` の catch を貫通)。binary は `Err` を返す。

## `tellg()` の罠

`BinTools::Read` がペイロードを最後まで読むと eofbit が立つ。`tellg()` は sentry が `good()` を見て failbit を立て **-1 を返す**。トレイラー無しのファイルで必ず踏む。`clear()` は `rdstate` だけを書き換え get ポインタを動かさないので、直後の `tellg()` が正しい位置を返す。

## 前置マジックが消したバグ

末尾マジックは **検索**だった。ペイロード末尾4バイトが偶然 `CDCL` になると、直前4バイトを個数として読み、ペイロードを `8 + N*16` バイト切り詰めて OCCT に渡す — **色を持たない正当な BRep が読めなくなる**。

前置マジックは確率を下げたのではなく検索を消した。`buf[consumed..]` はパーサが取らなかったバイトなので、定義上ペイロードではありえない。代わりに賭けているのは「`consumed` がペイロード終端ぴったり」の1点だけで、これはテストで固定できる(`trailer_begins_where_the_payload_ends`)。

## 能力の喪失

ASCII `.brep` は OCCT の `DRAWEXE save` が吐く既定形式。cadrum に食わせると `BrepReadFailed` になる。単なる改名ではないので CHANGELOG の Removed に書いた。
