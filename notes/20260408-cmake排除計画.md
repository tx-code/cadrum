# cmake排除計画: ccクレートによるOCCT直接コンパイル

## 背景

現在のbuild.rsはOCCTをソースからビルドする際に`cmake`クレート（= CMake CLI）に依存している。
CMakeを完全に排除し、`cc`クレート（cxx-build経由で既に利用可能）で直接コンパイルする。

## 現状分析

### CMakeがやっていること

1. `Standard_Version.hxx.in` → `Standard_Version.hxx` のテンプレート置換（バージョン番号のみ）
2. 各ツールキットの`FILES`マニフェストを読んで`.cxx`をコンパイル
3. スタティックライブラリにアーカイブ

### OCCTソースの構造

```
src/
├── TKernel/          ← ツールキット
│   ├── PACKAGES      ← パッケージ一覧（1行1パッケージ名）
│   ├── FILES         ← ツールキット直属ファイル（PCHヘッダ等）
│   └── EXTERNLIB     ← 依存関係（TK*名 or CSF_*システムライブラリ）
├── Standard/         ← パッケージ（TKernelのPACKAGESに記載）
│   └── FILES         ← ソースファイル一覧（1行1ファイル名、.cxxと.hxxが混在）
├── OSD/              ← パッケージ
│   └── FILES
...
```

2段階の構造: **ツールキット → パッケージ → ファイル**

### 生成ヘッダ

CMakeが生成するヘッダは **`Standard_Version.hxx` のみ**。

テンプレート `adm/templates/Standard_Version.hxx.in` の置換変数:
- `@OCC_VERSION_MAJOR@` → `7`
- `@OCC_VERSION_MINOR@` → `9`
- `@OCC_VERSION_MAINTENANCE@` → `3`
- `@SET_OCC_VERSION_DEVELOPMENT@` → 空（リリースビルド）
- `@OCCT_VERSION_DATE@` → ビルド日付

`str::replace`で十分に生成可能。

### CMake defineの`cc::Build`での対応

現在のcmake::Config設定を3カテゴリに分類する。

#### (A) USE_*=OFF → HAVE_*を定義しない

全て`OFF`なので、`cc::Build`では**何もしない**（`HAVE_*`を定義しなければ同等）。

| CMake設定 | 不使用のプリプロセッサ定義 |
|-----------|--------------------------|
| USE_FREETYPE=OFF | HAVE_FREETYPE |
| USE_FREEIMAGE=OFF | HAVE_FREEIMAGE |
| USE_TBB=OFF | HAVE_TBB |
| USE_RAPIDJSON=OFF | HAVE_RAPIDJSON |
| USE_DRACO=OFF | HAVE_DRACO |
| USE_OPENGL=OFF | HAVE_OPENGL_EXT |
| USE_GLES2=OFF | HAVE_GLES2_EXT |
| USE_D3D=OFF | HAVE_D3D |
| USE_XLIB=OFF | HAVE_XLIB |
| USE_VTK=OFF | HAVE_VTK |
| USE_OPENVR=OFF | HAVE_OPENVR |
| USE_FFMPEG=OFF | HAVE_FFMPEG |
| USE_TK=OFF | HAVE_TK |
| USE_TCL=OFF | (TCLスクリプト無効) |

→ `cc::Build`での対応: **不要（定義しないことが正解）**

#### (B) BUILD_LIBRARY_TYPE=Static → 定義が必要

| プリプロセッサ定義 | 効果 |
|-------------------|------|
| `OCCT_STATIC_BUILD` | `Standard_EXPORT`を空にする（DLLエクスポート抑制） |
| `OCCT_NO_PLUGINS` | プラグイン動的ロードを無効化 |

→ `cc::Build`での対応:
```rust
build.define("OCCT_STATIC_BUILD", None);
build.define("OCCT_NO_PLUGINS", None);
```

#### (C) BUILD_MODULE_*=ON/OFF → コンパイル対象の選択のみ

プリプロセッサ定義は設定されない。どのツールキットをコンパイルするかの選択。
`cc::Build`では対象ツールキットのFILESからソースを集めるだけで同等。

#### (D) グローバルコンパイル定義（CMakeが暗黙的に設定）

| 定義 | 条件 | `cc::Build`での対応 |
|------|------|---------------------|
| `_CRT_SECURE_NO_WARNINGS` | Windows | `.define("_CRT_SECURE_NO_WARNINGS", None)` |
| `_CRT_NONSTDC_NO_DEPRECATE` | Windows | `.define("_CRT_NONSTDC_NO_DEPRECATE", None)` |
| `UNICODE` | Windows | `.define("UNICODE", None)` |
| `_UNICODE` | Windows | `.define("_UNICODE", None)` |
| `_USE_MATH_DEFINES` | Windows | 既にbuild.rsで設定済み |

#### まとめ: `cc::Build`に必要なdefine一覧

```rust
// 全プラットフォーム共通
build.define("OCCT_STATIC_BUILD", None);
build.define("OCCT_NO_PLUGINS", None);

// Windows固有
if cfg!(target_os = "windows") {
    build.define("_CRT_SECURE_NO_WARNINGS", None);
    build.define("_CRT_NONSTDC_NO_DEPRECATE", None);
    build.define("UNICODE", None);
    build.define("_UNICODE", None);
    build.define("_USE_MATH_DEFINES", None);
}
// HAVE_*は全て未定義（USE_*=OFFと同等）
// BUILD_MODULE_*はソース選択で表現（defineではない）
```

## 実装計画

### Step 1: build.rsに`build_occt_with_cc`関数を追加

```rust
fn build_occt_with_cc(source_dir: &Path, install_prefix: &Path) -> (PathBuf, PathBuf) {
    // 1. Standard_Version.hxx を生成
    // 2. 全パッケージディレクトリをインクルードパスとして収集
    // 3. ツールキットごとにcc::Buildでコンパイル
    //    - FILES内の.cxxファイルを収集
    //    - 全パッケージディレクトリをインクルード
    //    - C++17, Release最適化
    // 4. ヘッダをinstall_prefix/incにコピー
}
```

### Step 2: ツールキットのコンパイル順序

依存関係に基づく順序（EXTERNLIBから導出）:

```
TKernel → TKMath → TKG2d → TKG3d → TKGeomBase → TKBRep
→ TKGeomAlgo → TKTopAlgo → TKPrim → TKBO → TKBool
→ TKShHealing → TKMesh → TKOffset
→ TKCDF → TKBin → TKCAF → TKLCAF
→ TKXSBase → TKDE → TKDECascade → TKXCAF → TKDESTEP
```

注: `cc::Build`はスタティックライブラリを生成するだけでリンクは最後に行われるので、
コンパイル順序の厳密性は不要。全ツールキットを1つの`cc::Build`にまとめることも可能。

### Step 3: インクルードパスの収集

```rust
// src/ 以下の全ディレクトリをインクルードパスに追加
// （OCCTのヘッダは #include <Standard.hxx> のようにパッケージ名なしで参照される）
for entry in fs::read_dir(source_dir.join("src")) {
    if entry.path().is_dir() {
        build.include(entry.path());
    }
}
// 生成ヘッダ用のディレクトリも追加
build.include(generated_inc_dir);
```

### Step 4: Cargo.toml変更

```diff
 [build-dependencies]
 cxx-build = "1"
 ureq = "3"
 libflate = "2"
 tar = "0.4"
-cmake = "0.1"
+cc = "1"
```

`cc`は`cxx-build`の依存に含まれるため、明示追加は任意だが明示する方が明確。

### Step 5: パッチ処理の維持

既存の`patch_occt_sources`（TKService依存除去）はそのまま維持。
`cc::Build`でコンパイルする前にパッチを適用する。

## 検討事項

### コンパイル時間

- 4,013個の.cxxファイルをコンパイルするため、CMake経由と同等の時間がかかる
- `cc::Build`は自動で並列コンパイル（`NUM_JOBS`環境変数で制御）
- 増分ビルドは`cc`クレートが`rerun-if-changed`で管理

### 1つのcc::Buildにまとめるか、ツールキットごとに分けるか

**案A: 1つにまとめる**
- 実装がシンプル
- 1つの巨大な`libocct.a`が生成される
- 増分ビルドの粒度が粗い（1ファイル変更で全再コンパイル…ただしOCCTソースは変更しない）

**案B: ツールキットごとに分ける**（推奨）
- 現在のリンク構造（`cargo:rustc-link-lib=static=TK*`）と整合
- `cc::Build::new().files(...).compile("TKernel")` で `libTKernel.a` が生成される
- デバッグ時にどのツールキットで問題が起きたか特定しやすい

### 環境依存（残るもの）

- **C++17対応コンパイラ**: MSVC (Windows) or GCC (Linux) — `cc`クレートが自動検出
- **それ以外**: なし（cmake不要、ninja不要、make不要）
