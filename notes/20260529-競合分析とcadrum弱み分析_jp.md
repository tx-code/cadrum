# 競合分析と cadrum の弱み分析

cadrum（statically-linked headless OCCT 8.0 を使う Rust CAD ライブラリ）の競合調査と、
求められているのに応えられていない機能／求められていないのに複雑性を増やしている機能の分析。
事実確認は README・`Cargo.toml`・`src/traits.rs`（全公開 API のソース）とウェブ出典に基づく。

関連: [OCCT/PureRust デュアルバックエンド計画](20260404-OCCT_PureRustデュアルバックエンド計画.md) /
[boolean は単体×単体のみ公開する方針](20260514-boolean演算は単体x単体のみ公開する方針.md)

---

## 1. 競合マップ

cadrum の立ち位置は「**BRep カーネル(OCCT)を Rust から、最小型・静的リンク・プリビルトで使う**」。

### A. Python / OCCT 系（機能の事実上の標準・最大の競合圏）
- **CadQuery** — OCCT(OCP bindings)上の fluent Python CAD。Workplane + Selector が中核。STEP/DXF/STL/3MF。機能の比較基準。
- **build123d** — CadQuery 派生を全面リファクタした後継。よりPythonic／algebraic。Python 圏の最新潮流。
- **pythonocc-core / OCP** — OCCT の 1:1 バインディング。cadrum が README で名指しで避けると言っている「クラス階層 1:1 露出」層。

### B. Rust / OCCT バインディング系（直接の同類・最重要競合）
- **opencascade-rs** (`bschwind`) — cxx.rs ベース。fillet/chamfer/loft/pipe/extrude/revolve、STEP/STL/SVG/DXF/KiCAD I/O。OCCT を **git submodule でソースビルド**。
- **opencascade-sys / occt-sys / tonarino/occt-sys** — OCCT 取得・sys 層。`occt-sys` は「静的ビルドして Rust 依存にする」点で cadrum の build.rs と同じ問題を別解法で解いている。
- **occara** — 別の小規模バインディング試み。

### C. Pure Rust カーネル系（OCCT 非依存＝配布は楽、機能は未成熟）
- **truck** (`ricosjp`) — フル Rust BRep カーネル。base/geometry/topology/modeling/meshalgo/stepio にクレート分割、NURBS、STEP I/O、WebGPU/WASM 志向。boolean/defeaturing/healing も標榜するが OCCT の堅牢性には遠い。
- **Fornjot** — 早期段階の純 Rust BRep。「数十年の OCCT には機能で全く及ばない」と自認。

→ cadrum の `pure` feature（`Cargo.toml` にスタブ、`traits.rs` がバックエンド非依存に書かれている）はこの C 領域を将来の自前/外部バックエンドで狙う布石。

### D. メッシュ/CSG 系（BRep ではない＝用途が別）
- **Manifold** — watertight メッシュ CSG カーネル(C++)。OpenSCAD の新エンジン。正確な BRep/STEP は保てない。
- **csgrs** — Rust の CSG（union/difference/intersection/xor）、OpenSCAD 風。dimforge エコシステム連携。
- **OpenSCAD** — スクリプト CAD のデファクト。CadQuery 系が「CGAL より OCCT が強力」と対比する相手。

## 2. CadQuery との差

**共通点**: OCCT を CAD カーネルに使う／fluent なメソッドチェーン／STEP・STL 出力／BRep。

| 観点 | CadQuery | cadrum |
|---|---|---|
| 言語 | Python（REPL/Jupyter 即時試行、メイカー向け） | Rust（静的型、組込/WASM/バックエンド向け） |
| 配布・依存 | OCP 依存が重い、Python 3.9–3.12 限定、ディストロ差で苦労 | **プリビルト OCCT 静的リンク**。`cargo build` 完結、`LD_LIBRARY_PATH` 不要、単一バイナリ出荷 |
| 中核抽象 | **Workplane**(2D 作業平面)+**Selector**(`">Z"`,`"\|X"` 等の文字列で面/辺/頂点を幾何条件選択) | 抽象を持たず **3 具象型 Solid/Edge/Face** にメソッド。選択は `iter_face().filter(closure)` の**手書き幾何条件** |
| 2D スケッチ | 強力(平面上で 2D 描画→push/cut) | **2D スケッチ層が無い**。`Edge::polygon`/`bspline` を直接 3D 座標で組む |
| フィーチャ参照の安定性 | Selector で「最上面」等を意味参照（パラメタ変化に強い） | `iter_history()` の `[post_id, src_id]` で boolean 越し面追跡（強力だが低レベル・自前） |
| 想定ユーザー | 対話的に手早くモデリング | アプリ組込で型安全に・配布容易に BRep を回す（例 katachiform.com の configurator） |

要点: **CadQuery＝人が対話的に手早く** / **cadrum＝アプリに組み込んで型安全・配布容易に**。
正面衝突でなく言語圏と配布形態でレイヤーが違う。cadrum が学べる最大点は **Workplane / Selector という「意味でのフィーチャ参照」抽象**。

## 3. cadrum の弱み分析

### 3-1. 機能ギャップ（求められているのに応えられていない）＋ 優先順位

`traits.rs` で公開 API を確認した結果のギャップ。優先度は **需要の高さ × 実装容易度（OCCT に既存 API があるか）× cadrum 哲学(型を減らす/意図を見せる)との整合** で評価。

| # | ギャップ | 需要 | 実装容易度 | 優先度 | 根拠/備考 |
|---|---|---|---|---|---|
| 1 | **`revolve`(回転体)** が無い | 高 | 高 (`BRepPrimAPI_MakeRevol`) | **最優先** | primitive→extrude→sweep→loft→bspline と揃うのに最も基本的な回転押出が欠落。opencascade-rs にはある。`Solid::extrude` と同じシグネチャ感で追加可能 |
| 2 | **意味ベースのフィーチャ選択(Selector 相当)** が無い | 高 | 中 | **高** | 現状 `iter_edge().filter(\|e\| (p.z-h).abs()<1e-6)` の座標ベタ書き(10_fillet/08_shell が実例)。CadQuery の `">Z"` 相当。cadrum 哲学に沿い、文字列でなく小さな述語型(例 `Extreme::Max(Axis::Z)` / `Parallel(axis)`)を `iter_face`/`iter_edge` の引数に。型で意図を見せられる |
| 3 | **2D スケッチ/Workplane 抽象** が無い | 高 | 中〜高 | 中 | プロファイルを常に 3D 座標で書かせる。「平面+2D 座標→edge 群」の薄いヘルパだけでも実用度が上がる。型を増やすので AGENTS.md 方針と要トレードオフ |
| 4 | **`offset` / `draft`(抜き勾配) / 可変半径フィレット** | 中 | 中 (OCCT 既存) | 中 | 製造系需要。`fillet_edges` を可変半径対応に拡張すると関数を増やさず済む |
| 5 | **DXF 入力(2D ベクタ→プロファイル)** | 中 | 中 | 中 | opencascade-rs は DXF/SVG/KiCAD まで。cadrum は STEP/BRep/STL/SVG(出力)/PNG。プロファイル供給源として DXF 読込が効く |
| 6 | **アセンブリ/名前付きパーツ** | 中 | 中 (XDE 拡張) | 低〜中 | `Vec<Solid>` は平坦な集合。STEP のアセンブリ階層/部品名/配置の保持は産業用途で期待。color は既に XDE 実装済なので延長で届く |

### 3-2. 複雑性削減の精査（求められていないのに在って複雑性を増やす疑い）

AGENTS.md「関数/型/モジュールは減らす方向を優先」に照らし、API 面を増やしている要素を「価値 vs 維持/学習コスト」で精査。

| 要素 | 状況 | 削減/整理の検討 |
|---|---|---|
| **`Solid::bspline`(2D 制御点グリッド→ソリッド)** | 09_bspline はステラレータ等ニッチ。`point(i,j)` クロージャ引数で API が重い | 汎用 CAD 需要は薄い。loft/sweep で代替可能かを検証し、可能なら example だけ残して公開 API から外す候補。**最も削減効果が見込める** |
| **`ProfileOrient` 4 variant (Fixed/Torsion/Up/Auxiliary)** | 表現力は高いが `Auxiliary(aux_spine)` は上級。初学者に選択が難しい | デフォルト(`Up`/`Torsion`)＋上級オプションへの整理。doc は既に充実しているので「廃止」より「推奨フローの明示」で十分かも |
| **BRep text + binary の二系統 I/O** | STEP に加え BRep が 2 系統 = 計 3 入出力系統 | OCCT ネイティブ往復という価値はあるが、ユーザー主用途がどれか精査。text/binary 片方に寄せれば read/write 4 関数を削減できる |
| **`half_space` primitive** | shell の補助としては使うが直接利用頻度は低い | primitive 一覧から boolean 補助 helper に格下げ検討の余地 |
| **`pure` feature（純 Rust バックエンド）** | `Cargo.toml` にスタブ・未実装。`traits.rs` 全体がこの抽象のためバックエンド非依存に書かれている | **戦略判断が必要**(§3-3)。truck/Fornjot の成熟度を見るに自前 pure 実装の維持コストは極大。「やらない」と決めれば `traits.rs` の関連型による間接化を一段単純化できる。既存検討: [20260404-OCCT_PureRustデュアルバックエンド計画](20260404-OCCT_PureRustデュアルバックエンド計画.md) |

補足: boolean を `Compound` に載せず `SolidStruct` の単体×単体のみ公開する設計（[20260514-boolean演算は単体x単体のみ公開する方針](20260514-boolean演算は単体x単体のみ公開する方針.md)）や、`clean` を `Compound` に載せない判断は、**逆に「罠を増やさないための正しい複雑性削減」**として機能しており、評価できる。

### 3-3. 配布戦略の比較（cadrum の最大の差別化点）

cadrum のコア主張は機能ではなく**配布**。ここは明確な勝ち筋。

| 軸 | cadrum | opencascade-rs (Rust 同類) | truck (純 Rust) |
|---|---|---|---|
| OCCT 入手 | **プリビルト tarball を build.rs が DL → 静的リンク** | OCCT を git submodule で**毎回ソースビルド**(C++17/CMake、長時間、CMP0054 等の既知ビルド失敗報告あり) | OCCT 不要(純 Rust) |
| `cargo build` 体験 | 対応4ターゲットは即通る | submodule fetch + 長い C++ ビルド or 手動 OCCT インストール + `DEP_OCCT_ROOT` | 即通る |
| 出荷物 | 単一静的バイナリ、動的 lib なし | 同上(リンク自体は可) | 単一バイナリ |
| 機能成熟度 | OCCT 由来＝高 | OCCT 由来＝高 | 低(boolean/fillet の堅牢性が未成熟) |
| WASM/組込 | 静的リンク＋最小型で有利。`write_multiview_png` 等 LLM 駆動設計ループ向け機能も布石 | 可能だが OCCT ビルドが重い | 純 Rust で最も身軽だが機能不足 |

**含意**: cadrum は「OCCT の機能成熟度」と「純 Rust 並みの配布容易さ」を**プリビルト静的リンクで両取り**しようとしている唯一の Rust プロジェクト。これが核心的差別化。
- 強化すべき: プリビルト対応ターゲットの拡大（現状 Linux/Windows × x86_64 + 一部 aarch64）。特に **wasm32 と macOS** はメイカー/組込需要が大きい弱点。
- リスク: プリビルト tarball のホスティング/バージョン追従が属人的になりやすい。OCCT 8.x 追従の CI 自動化が継続性の鍵。
- `pure` feature は「配布容易さ」をさらに突き詰める方向だが、機能成熟度を捨てることになるため §3-2 の通り戦略判断が必要。

## 4. コレクション系の複雑性削減（実施記録: `Compound` / `Wire` 削除）

§3-2 で「求められていないのに複雑性を増やす疑い」として挙げた候補のうち、最初に
着手したのが **コレクション抽象トレイト `Compound` / `Wire` の撤廃**。2026-05-29 に実施。

### 動機（なぜ削れたか）
- `Compound` / `Wire` は「Solid/Edge の集まり」を表す pub トレイトだが、実装は
  `Vec<T>` と `[T; N]` の **2 型だけ**。`&[T]`・イテレータ・`HashSet` 等には無い。
  これは `map_elem(self, f) -> Self`（transform が `Self` を返す設計）が
  **所有・同形再構築可能なコンテナ**を要求するため構造的に 2 型へ縛られていたもので、
  「2 つだけが特別」という恣意性がユーザーの違和感の正体だった。
- `impl Compound for Solid` / `impl Wire for Edge` の `iter_elem` / `map_elem` は
  **`panic!`** する歪んだ実装で、codegen 経由で `Solid::iter_elem()` /
  `Edge::map_elem()` という **panic する pub メソッド（footgun）** として外部露出していた。
- 両トレイトは **境界(bound)として一度も使われていない**（`T: Compound` 等ゼロ）。
  sweep/loft/extrude/boolean/I-O は全て `impl IntoIterator<Item = &Edge>` / `&Solid`
  を取る。つまり「メソッドの運び屋」でしかなく、削除してもジェネリックなシグネチャは壊れない。
- 単体メソッドは全て backend で override された FFI 直叩きで、トレイト default
  （集約ロジック）に依存していなかった。

### 方針（3 層への分解と判断）
トレイトに同居していた 3 つの役割を分離して処理した:

| 層 | 中身 | 判断 |
|---|---|---|
| **L1 単体語彙** | `Solid` の volume/center/translate…、`Edge` の start_point/project… | `SolidStruct` / `EdgeStruct` に吸収（両方 `: Transform`）。`Transform` は単体共有トレイトとして存続。ユーザー影響ゼロ |
| **L2 コレクション transform** | `vec.translate()` 等 | **廃止**。`vec.into_iter().map(\|s\| s.translate(v)).collect()` / 配列は `arr.map(...)`。AGENTS.md「sugar で意図を隠さない」に忠実化 |
| **L3 コレクション集約** | `vec.volume()`=Σ, bbox union, 加重 center, inertia, wire project/approx | **全廃**（静的関数も作らず）。`a.inertia() + b.inertia()`、`solids.iter().map(\|s\| s.volume()).sum()` をユーザーが書く |

> `map_elem` が消えたことで「コンテナが 2 型に縛られる」制約自体が消滅し、
> コレクションは標準イテレータで一様に扱えるようになった（§3-2 の違和感が根本解決）。

### 結果
- `src/traits.rs` から `Compound` / `Wire` トレイトと `Vec<T>`/`[T;N]` 向け
  `Transform`/`Compound`/`Wire` impl（計 6 ブロック）を削除。トレイト総数 5→3。
- `panic!` する `iter_elem` / `map_elem` の pub 露出が消滅。
- `examples/codegen.rs` は **無変更**で追従（`SolidStruct`/`EdgeStruct` の supertrait を
  `Transform` に変えるだけで生成 forwarder の参照先が自動で切り替わる）。再生成して `no diff`。
- 例 3 本（02/07/101）とテスト 3 本（mass_properties/edge/svg）をイテレータ慣用句へ移行。
  集約系テストは「単体 FFI 値＋ユーザー側合成」を検証する形に作り替え（例: inertia の加法性テスト）。
- 検証: `cargo build`（default / color オフ）0 warn、`cargo test` 17 バイナリ pass、
  doctest 19 pass。ブランチ `remove/compound_wire`。

### 評価と次の候補
今回の削除は §3-2 で挙げた他候補（`Solid::bspline` のニッチ性、`ProfileOrient` の
4 variant、BRep text/binary 二系統、`half_space` primitive）へ着手する前段として、
「コレクション抽象を持たない」という土台を確定させた。逆に boolean を単体×単体のみ
公開する設計（[20260514](20260514-boolean演算は単体x単体のみ公開する方針.md)）と同様、
**API 面を増やさず罠を減らす**方向に一貫している。
