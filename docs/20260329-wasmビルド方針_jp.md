# OCCT wasm32ビルド方針

## 目標

chijinのコア幾何演算（Shape構築・Boolean演算・メッシング）を
`wasm32-unknown-unknown` ターゲットで動作させる。
Emscripten不要、libc不要、ブラウザ/Node.jsから直接利用可能なwasmモジュールを生成する。

## 前提: なぜ可能か

chijinが使うOCCTの機能は本質的にメモリ演算で閉じている:

| 機能 | OS依存 | 備考 |
|------|--------|------|
| Shape構築 (Box, Cylinder等) | なし | 純粋なメモリ演算 |
| Boolean演算 (Fuse, Cut, Common) | なし | 純粋なメモリ演算 |
| Meshing | なし | 純粋なメモリ演算 |
| STEP/BREP I/O | なし | RustReader/RustWriter経由で既にファイル不要 |
| Translate, Rotate, Scale | なし | 純粋なメモリ演算 |

OS依存はTKernelのOSD層に集中しており、chijinの機能パスからは到達しない。

## アーキテクチャ

```
wasm-bindgen (JS ↔ Rust)
  ↓
chijin (Rust)
  ↓ cxx FFI
OCCT (静的ライブラリ、OS依存部スタブ化)
  ↓
libc++ freestanding (libcxx、スレッド/ファイル/locale無効)
  ↓
最小限のCシンボル (malloc/free/memcpy/abort)
  ↓ Rustランタイムが提供
wasm linear memory
```

## 3つの作業

### 作業1: libc++ freestandingビルド

LLVMの libc++ (libcxx) + libc++abi (libcxxabi) を
wasm32ターゲットでクロスコンパイルする。

```cmake
cmake -G Ninja llvm-project/runtimes \
  -DCMAKE_C_COMPILER=clang \
  -DCMAKE_CXX_COMPILER=clang++ \
  -DCMAKE_C_COMPILER_TARGET=wasm32-unknown-unknown \
  -DCMAKE_CXX_COMPILER_TARGET=wasm32-unknown-unknown \
  -DLLVM_ENABLE_RUNTIMES="libcxx;libcxxabi" \
  -DLIBCXX_ENABLE_SHARED=OFF \
  -DLIBCXX_ENABLE_STATIC=ON \
  -DLIBCXX_ENABLE_FILESYSTEM=OFF \
  -DLIBCXX_ENABLE_THREADS=OFF \
  -DLIBCXX_ENABLE_MONOTONIC_CLOCK=OFF \
  -DLIBCXX_ENABLE_RANDOM_DEVICE=OFF \
  -DLIBCXX_ENABLE_LOCALIZATION=OFF \
  -DLIBCXX_ENABLE_UNICODE=OFF \
  -DLIBCXX_ENABLE_WIDE_CHARACTERS=OFF \
  -DLIBCXX_ENABLE_EXCEPTIONS=OFF \
  -DLIBCXXABI_ENABLE_EXCEPTIONS=OFF \
  -DLIBCXXABI_ENABLE_THREADS=OFF \
  -DLIBCXXABI_USE_LLVM_UNWINDER=OFF
```

無効化する理由:
- `FILESYSTEM=OFF` — wasm にファイルシステムなし
- `THREADS=OFF` — wasm はシングルスレッド（SharedArrayBuffer未使用時）
- `MONOTONIC_CLOCK=OFF` — `clock_gettime` なし
- `RANDOM_DEVICE=OFF` — `/dev/urandom` なし
- `LOCALIZATION=OFF` — ロケール機能不要、POSIX依存を排除
- `EXCEPTIONS=OFF` — wasm例外は未成熟、OCCT側でも無効化

libc++が要求する最小限のCシンボル:
- `malloc` / `free` / `realloc` / `calloc` → Rustの`dlmalloc`が提供
- `memcpy` / `memmove` / `memset` / `memcmp` / `strlen` → `compiler-builtins` crateが提供
- `abort` → `__builtin_trap` またはRust側の `__rust_abort`

これらは全てRustのwasmランタイムがエクスポートするため、
追加のlibc実装は不要。

### 作業2: OCCTソースのスタブ化

build.rsの既存 `patch_occt_sources` 機構を拡張し、
OS依存ファイルのメソッドボディを空化する。

#### スタブ対象 (TKernel — OSD層)

| ファイル | 依存 | 方針 |
|---------|------|------|
| `OSD_File.cxx` | open/read/write/close | 空化 |
| `OSD_Path.cxx` | ファイルパス操作 | 空化 |
| `OSD_Signal.cxx` | signal, MessageBoxA | 空化 |
| `OSD_SharedLibrary.cxx` | dlopen/LoadLibrary | 空化 |
| `OSD_Thread.cxx` | pthread_create | 空化 |
| `OSD_Environment.cxx` | getenv/setenv | 空化 |
| `OSD_Directory.cxx` | mkdir/opendir | 空化 |
| `OSD_FileSystem.cxx` | fopen/fclose | 空化 |
| `OSD_Parallel.cxx` | TBB/OpenMP | 空化 |
| `OSD_MemInfo.cxx` | /proc/meminfo, sysinfo | 空化 |

#### スタブ対象 (TKernel — その他)

| ファイル | 依存 | 方針 |
|---------|------|------|
| `Standard_ErrorHandler.cxx` | setjmp/longjmp, signals | 空化 |
| `FSD_File.cxx` | ファイルI/O | 空化 |
| `Plugin_MapOfFunctions.cxx` | 動的ロード | 空化 |

#### スタブ不要 (メモリ演算のみ)

- `Standard_MMgrOpt.cxx` — malloc/freeラッパー → Rustのallocatorで動作
- `NCollection_*` — 純粋なコンテナ・アルゴリズム
- `gp_*`, `TopLoc_*`, `TopoDS_*` — 幾何データ構造

#### スタブ手順

1. `cargo build --target wasm32-unknown-unknown` でリンクエラーを収集
2. 未解決シンボルからソースファイルを特定 (`nm` / `grep`)
3. `patch_occt_sources` に追加して空化
4. リンクエラーが消えるまで繰り返し

`stub_all_top_level_bodies` は全メソッドを空化するため、
1ファイルに必要な関数と不要な関数が混在する場合は
ファイル単位で空化してからリンクエラーで必要な関数だけ戻す。

### 作業3: build.rs のwasm対応

現在のbuild.rsはwasm32で即returnしているが、
これをOCCTのCMakeビルド + スタブ化を行うように変更する。

```rust
if target_arch == "wasm32" {
    let (occt_include, occt_lib_dir) = build_occt_wasm(&out_dir, &manifest_dir);
    link_occt_libraries_wasm(&occt_include, &occt_lib_dir);
    return;
}
```

CMake設定のwasm差分:
```cmake
-DCMAKE_SYSTEM_NAME=Generic
-DCMAKE_C_COMPILER=clang
-DCMAKE_CXX_COMPILER=clang++
-DCMAKE_C_COMPILER_TARGET=wasm32-unknown-unknown
-DCMAKE_CXX_COMPILER_TARGET=wasm32-unknown-unknown
-DCMAKE_C_FLAGS="--sysroot=... -fno-exceptions"
-DCMAKE_CXX_FLAGS="--sysroot=... -fno-exceptions -nostdinc++ -isystem <libc++/include>"
-DBUILD_MODULE_Visualization=OFF
-DBUILD_MODULE_Draw=OFF
-DUSE_FREETYPE=OFF
-DUSE_TBB=OFF
# ... (既存のbundledと同様のOFF設定)
```

## lib.rs の cfg ゲート方針

wasmビルドでもOCCT FFIモジュールを有効化する（現在は除外）。
段階的に:

1. まずスタブ化OCCTでcxx bridgeがリンクを通すことを確認
2. 次にwasm-bindgenでJS向けAPIを公開
3. 最終的に `#[cfg(target_arch = "wasm32")]` は不要になる

## 今の自作sysrootとの関係

`wasm-sysroot/` は libc++ freestanding に置き換わる。
自作ヘッダは開発中の動作確認用として残すが、
最終的にはlibc++の方が信頼性・互換性で優位。

| | 自作sysroot (現状) | libc++ freestanding (最終形) |
|--|---|---|
| std::string | SSO無し簡易実装 | 完全実装 |
| std::vector | 例外安全性なし | 完全実装 |
| iostream | no-op | バッファ付きstreambuf |
| テスト済み | cxxコンパイルのみ | LLVM CIで継続テスト |
| OCCT対応 | 未検証 | 検証予定 |

## リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| OCCTのスタブ化漏れ | リンクエラー | エラー駆動で追加 |
| OCCTが内部でファイルパスを仮定 | 実行時エラー | stream I/O経路のみ使用 |
| libc++のCシンボル不足 | リンクエラー | wasm-sysrootのC関数宣言を流用 |
| wasmバイナリサイズ | 数MB超 | wasm-opt + LTO + 未使用コード除去 |
| STEP readerのスキーマファイル参照 | 読み込み失敗 | STEPCAFControl_Readerの内部パスを調査 |

## JS向けAPI設計: wasm-bindgenの制約と対策

### wasm-bindgenが扱える型

| OK | NG |
|----|----|
| プリミティブ (`i32`, `f64`, `bool`) | ジェネリクス (`Vec<T>` でTがカスタム型) |
| `String`, `&str` | トレイトオブジェクト (`dyn Shape`) |
| `Vec<u8>`, `Vec<f64>` 等 (数値Vec) | `Vec<Solid>` (カスタム型のVec) |
| `#[wasm_bindgen]` 付きstruct | `&[Solid]` (カスタム型スライス) |
| `JsValue` | トレイトメソッド (`impl Shape for Vec<Solid>`) |

chijinの `Shape` トレイト (`impl Shape for Vec<Solid>`) は
`union`, `subtract`, `faces`, `translate` 等をスライス `&[Solid]` に対して
提供するが、これらはwasm-bindgenの境界を直接越えられない。

### 方針: opaque handle パターン

`Vec<Solid>` をopaque handle (`Scene` struct) に包み、
プリミティブ引数のメソッドだけを `#[wasm_bindgen]` で公開する。

```rust
// src/wasm.rs

use wasm_bindgen::prelude::*;
use crate::solid::Solid;
use crate::shape::{Boolean, Shape};

/// JS側に公開するopaque handle。
/// 内部の Vec<Solid> はJS側からは不透明。
#[wasm_bindgen]
pub struct Scene {
    solids: Vec<Solid>,
}

#[wasm_bindgen]
impl Scene {
    // ==================== コンストラクタ ====================

    #[wasm_bindgen(constructor)]
    pub fn new() -> Scene {
        Scene { solids: vec![] }
    }

    // ==================== プリミティブ追加 ====================

    pub fn add_box(&mut self,
        x1: f64, y1: f64, z1: f64,
        x2: f64, y2: f64, z2: f64,
    ) {
        self.solids.push(Solid::box_shape(
            glam::dvec3(x1, y1, z1),
            glam::dvec3(x2, y2, z2),
        ));
    }

    pub fn add_cylinder(&mut self,
        px: f64, py: f64, pz: f64,
        dx: f64, dy: f64, dz: f64,
        radius: f64, height: f64,
    ) {
        self.solids.push(Solid::cylinder(
            glam::dvec3(px, py, pz),
            glam::dvec3(dx, dy, dz),
            radius, height,
        ));
    }

    // ==================== Boolean演算 ====================
    // 新しいSceneを返す（元は消費しない）

    pub fn union(&self, other: &Scene) -> Result<Scene, JsError> {
        let result = Boolean::union(&self.solids, &other.solids)
            .map_err(|e| JsError::new(&e.to_string()))?;
        Ok(Scene { solids: result.into() })
    }

    pub fn subtract(&self, other: &Scene) -> Result<Scene, JsError> {
        let result = Boolean::subtract(&self.solids, &other.solids)
            .map_err(|e| JsError::new(&e.to_string()))?;
        Ok(Scene { solids: result.into() })
    }

    pub fn intersect(&self, other: &Scene) -> Result<Scene, JsError> {
        let result = Boolean::intersect(&self.solids, &other.solids)
            .map_err(|e| JsError::new(&e.to_string()))?;
        Ok(Scene { solids: result.into() })
    }

    // ==================== 変換 ====================

    pub fn translate(&self, tx: f64, ty: f64, tz: f64) -> Scene {
        Scene {
            solids: self.solids.translate(glam::dvec3(tx, ty, tz)),
        }
    }

    pub fn rotate(&self,
        ox: f64, oy: f64, oz: f64,
        dx: f64, dy: f64, dz: f64,
        angle: f64,
    ) -> Scene {
        Scene {
            solids: self.solids.rotate(
                glam::dvec3(ox, oy, oz),
                glam::dvec3(dx, dy, dz),
                angle,
            ),
        }
    }

    // ==================== メッシュ出力 ====================
    // JS側ではTypedArrayとして受け取る

    /// 頂点座標を flat [x,y,z, x,y,z, ...] で返す
    pub fn mesh_vertices(&self, tolerance: f64) -> Vec<f64> {
        let mesh = self.solids.mesh(tolerance);
        mesh.vertices // Vec<f64> — wasm-bindgenがFloat64Arrayに変換
    }

    /// 三角形インデックスを flat [i,j,k, i,j,k, ...] で返す
    pub fn mesh_indices(&self, tolerance: f64) -> Vec<u32> {
        let mesh = self.solids.mesh(tolerance);
        mesh.indices // Vec<u32> — wasm-bindgenがUint32Arrayに変換
    }

    /// 法線ベクトルを flat [nx,ny,nz, ...] で返す
    pub fn mesh_normals(&self, tolerance: f64) -> Vec<f64> {
        let mesh = self.solids.mesh(tolerance);
        mesh.normals
    }

    // ==================== STEP I/O (バイト列) ====================

    /// STEPファイルをバイト列から読み込み
    pub fn from_step(data: &[u8]) -> Result<Scene, JsError> {
        let solids = crate::read_step(&mut &data[..])
            .map_err(|e| JsError::new(&e.to_string()))?;
        Ok(Scene { solids })
    }

    /// STEPファイルをバイト列として書き出し
    pub fn to_step(&self) -> Result<Vec<u8>, JsError> {
        let mut buf = Vec::new();
        crate::write_step(&self.solids, &mut buf)
            .map_err(|e| JsError::new(&e.to_string()))?;
        Ok(buf) // Vec<u8> — wasm-bindgenがUint8Arrayに変換
    }

    // ==================== 情報取得 ====================

    pub fn solid_count(&self) -> usize {
        self.solids.len()
    }

    pub fn is_empty(&self) -> bool {
        self.solids.is_empty()
    }

    pub fn volume(&self) -> f64 {
        self.solids.iter().map(|s| s.volume()).sum()
    }
}
```

### JS側の使用例

```js
import init, { Scene } from './wasm/chijin.js';

await init();

// プリミティブ作成
const box = new Scene();
box.add_box(0, 0, 0, 1, 1, 1);

const cyl = new Scene();
cyl.add_cylinder(0.5, 0.5, -0.5, 0, 0, 1, 0.3, 2);

// Boolean演算
const result = box.subtract(cyl);  // 箱から円柱をくり抜き

// メッシュ取得 → Three.js等で描画
const vertices = result.mesh_vertices(0.1);  // Float64Array
const indices  = result.mesh_indices(0.1);   // Uint32Array
const normals  = result.mesh_normals(0.1);   // Float64Array

// STEP入出力
const stepBytes = result.to_step();          // Uint8Array
const loaded = Scene.from_step(stepBytes);

// メモリ解放 (GCに任せることも可能だがexplicitが推奨)
box.free();
cyl.free();
```

### 型変換の対応表

| Rust (Scene メソッド) | JS (wasm-bindgen変換後) |
|---|---|
| `-> Vec<f64>` | `Float64Array` |
| `-> Vec<u32>` | `Uint32Array` |
| `-> Vec<u8>` | `Uint8Array` |
| `-> String` | `string` |
| `-> Result<Scene, JsError>` | `Scene` (失敗時はthrow) |
| `&[u8]` 引数 | `Uint8Array` |
| `f64` 引数 | `number` |
| `&Scene` 引数 | `Scene` (参照渡し) |

### 設計判断

- **Sceneは不変スタイル**: Boolean演算・変換は新しいSceneを返す。
  JS側でイミュータブルに扱えるため、状態管理がシンプル。
- **meshは毎回計算**: キャッシュしない。呼び出し側でキャッシュする方がwasmメモリ効率が良い。
- **エラーはJsError**: Rust側の `Error` を文字列変換してJSのExceptionに変換。
- **glam::DVec3は境界を越えない**: f64×3のプリミティブ引数に分解。
  JS側の利便性はTypeScript型定義 (.d.ts) でカバーする。

## 成功基準

1. `wasm-pack build` がリンクまで通る
2. wasm上で `Scene::add_box()` → `mesh_vertices()` が動作する
3. wasm上でBoolean演算 (`union`, `subtract`, `intersect`) が動作する
4. wasm上でSTEP読み書き（バイト列経由）が動作する
5. Three.js等のWebGLレンダラでメッシュが描画できる
